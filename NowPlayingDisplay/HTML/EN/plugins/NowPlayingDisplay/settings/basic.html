The plugin manages both SqueezeLite and the FFT helper directly. Two small system-level steps are needed once (both require root): load the snd-dummy kernel module at boot, and open the Dummy ALSA card's permissions via a udev rule so the LMS user can spawn squeezelite against it. The udev rule only affects the virtual Dummy card — no other audio hardware is touched.

One-time setup (any Linux):

PLUGIN=[% pluginBaseDir | html %]

# 1. Make snd-dummy load at boot (it's already in every distro's stock kernel).
sudo cp $PLUGIN/Service/snd-dummy.conf /etc/modules-load.d/snd-dummy.conf
sudo modprobe snd-dummy

# 2. Open the Dummy ALSA card's /dev/snd nodes so the LMS user can access them.
#    Only affects the virtual Dummy card, not any real audio hardware.
sudo cp $PLUGIN/Service/99-snd-dummy-permissions.rules /etc/udev/rules.d/
sudo udevadm control --reload
sudo udevadm trigger

# 3. Install apt deps (squeezelite + python3 + numpy) if you don't have them.
sudo apt install -y squeezelite python3 python3-numpy   # Debian/Ubuntu/DietPi
# sudo apk add squeezelite python3 py3-numpy            # Alpine
# sudo pacman -S squeezelite python python-numpy        # Arch
# sudo dnf install squeezelite python3 python3-numpy    # Fedora

That's it — turn on "Server-side visualizer" above and the plugin handles spawning both processes.

The default config runs SqueezeLite as:

squeezelite -n Visualizer -m [% prefs.vizPlayerMac | html %] -o plughw:CARD=Dummy,DEV=0 -v -s 127.0.0.1

Why a real SqueezeLite (and not a UPnPBridge-style approach)? UPnPBridge uses a custom fork that doesn't open ALSA — it just forwards audio to network renderers. We need the real binary because the -v shared-memory buffer (read by our FFT helper) is populated by SqueezeLite's audio output thread, which needs ALSA as a clocking source. The Dummy card gives us that timing without any actual sound playback. See $PLUGIN/Service/README.md for full notes.
